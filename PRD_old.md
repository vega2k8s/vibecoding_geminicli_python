# PRD: 업무 말투 변환기 (Biz-Talk Converter)

---

## 1. 개요 (Overview)

### 1.1. 제품 비전
"누구나, 어떤 상황에서든, 프로페셔널한 비즈니스 커뮤니케이션을 손쉽게 할 수 있도록 돕는다."

### 1.2. 배경 및 문제점
많은 직장인이 이메일, 메신저 등 디지털 툴로 소통하는 과정에서 적절한 어투나 표현을 고민하느라 불필요한 시간을 소모합니다. 특히 상대방(상사, 동료, 고객)과의 관계나 상황의 민감도에 따라 표현 방식이 달라져야 하므로, 이는 상당한 정신적 피로감을 유발합니다. 잘못된 표현은 오해를 낳고 협업의 효율을 떨어뜨리기도 합니다.

### 1.3. 목표
*   **핵심 목표:** 사용자가 입력한 핵심 정보(키워드)를 수신자 및 상황에 맞는 정중하고 명확한 비즈니스 메시지로 자동 변환하여 제공한다.
*   **세부 목표:**
    *   메시지 작성에 드는 시간을 50% 이상 단축시킨다.
    *   상황에 맞지 않는 표현 사용으로 인한 커뮤니케이션 실패율을 감소시킨다.
    *   사회초년생이나 비즈니스 작문에 익숙하지 않은 사용자에게 가이드 역할을 한다.

## 2. 사용자 및 시장

### 2.1. 타겟 사용자
*   비즈니스 커뮤니케이션에 어려움을 겪는 사회초년생
*   다양한 관계의 사람들과 효율적으로 소통해야 하는 모든 직장인
*   고객 응대, 사내 보고 등 특정 목적의 글쓰기를 자주 하는 실무자

### 2.2. 수신자 페르소나 (변환 대상)
본 서비스는 메시지를 받는 상대를 아래와 같이 세 그룹으로 정의하고, 각 그룹에 최적화된 결과물을 제공합니다.

*   **상사 (Upward):**
    *   **특징:** 보고의 명확성, 격식, 신뢰성을 중시함.
    *   **결과물 톤앤매너:** 정중한 격식체. 결론부터 명확하게 제시하는 보고 형식.
*   **타팀 동료 (Lateral):**
    *   **특징:** 협업의 원활함, 요청의 명확성을 중시함.
    *   **결과물 톤앤매너:** 친절하고 상호 존중하는 어투. 요청 사항과 마감 기한을 명확히 전달하는 협조 요청 형식.
*   **고객 (External):**
    *   **특징:** 서비스의 신뢰도, 친절함, 문제 해결을 중시함.
    *   **결과물 톤앤매너:** 극존칭을 사용하며, 전문성과 서비스 마인드를 강조. 안내, 공지, 사과 등의 목적에 부합하는 형식.

## 3. 기능 요구사항

### 3.1. MVP 핵심 기능
| 기능 ID | 기능명 | 상세 설명 | 우선순위 |
| :--- | :--- | :--- | :--- |
| F-001 | **멀티 페르소나 변환** | 사용자는 변환을 원하는 메시지의 수신자를 '상사', '동료', '고객' 중에서 선택할 수 있다. 선택된 페르소나에 따라 메시지의 말투, 형식, 단어 선택이 최적화되어 변환된다. | **High** |
| F-002 | **키워드 기반 입력** | 사용자는 '누가(Who)', '언제(When)', '무엇을(What)', '왜(Why)' 등 핵심 키워드 위주로 간결하게 정보를 입력할 수 있는 입력창을 제공받는다. 완벽한 문장을 입력하지 않아도 시스템이 이를 이해하고 완전한 문장으로 생성해야 한다. | **High** |
| F-003 | **결과 확인 및 복사** | 변환된 메시지는 별도의 결과창에 표시된다. 사용자는 '복사하기' 버튼을 클릭하여 생성된 메시지를 클립보드에 쉽게 복사할 수 있어야 한다. | **High** |
| F-004 | **기본 UI/UX** | 사용자가 기능을 쉽게 이해하고 사용할 수 있도록 간결한 UI(입력창, 라디오버튼/선택박스, 결과창, 변환 버튼)를 제공한다. | **Medium** |

## 4. 비기능 요구사항

### 4.1. 기술 스택
*   **Frontend:** Vanilla JS, HTML, CSS (별도 프레임워크 도입 최소화)
*   **Backend:** Python Flask
*   **AI Model:** Google Gemini API
*   **Deployment:** Firebase (Hosting & Functions)

### 4.2. 성능
*   **응답 시간:** 사용자로부터 변환 요청을 받은 후 3초 이내에 결과물을 제공해야 한다.

### 4.3. 사용자 경험 (UX)
*   직관적이고 단순해야 하며, 별도의 학습 없이 즉시 사용 가능해야 한다.
*   반응형 디자인을 적용하여 모바일 및 데스크톱 환경 모두에서 사용성이 보장되어야 한다.

## 5. 화면 디자인 및 스타일 (Screen Design and Style)

### 5.1. 디자인 원칙
*   **단순함 (Simplicity):** 복잡하지 않고 직관적인 레이아웃과 요소를 사용하여 사용자가 핵심 기능에 집중할 수 있도록 한다.
*   **일관성 (Consistency):** 모든 페이지와 요소에서 통일된 디자인 언어, 색상, 타이포그래피를 사용하여 일관된 사용자 경험을 제공한다.
*   **가독성 (Readability):** 텍스트의 크기, 색상 대비, 줄 간격 등을 고려하여 정보의 가독성을 높인다.

### 5.2. 레이아웃
*   **입력 섹션:** 사용자가 핵심 키워드를 입력하고 수신자 페르소나를 선택하는 영역. 명확하게 구분된 입력 필드와 라디오 버튼 또는 드롭다운 메뉴를 포함한다.
*   **결과 섹션:** 변환된 메시지가 표시되는 영역. 복사 버튼과 함께 결과 텍스트를 명확하게 보여준다.
*   **반응형 디자인:** 다양한 디바이스(데스크톱, 태블릿, 모바일)에서 최적의 사용성을 제공하기 위해 반응형 웹 디자인을 적용한다.

### 5.3. 시각적 스타일
*   **색상:** 차분하고 전문적인 느낌을 주는 색상 팔레트(예: 흰색, 회색 계열, 강조색으로 단색 포인트)를 사용한다.
*   **폰트:** 가독성이 높은 웹 표준 폰트(예: Noto Sans KR, Pretendard 등)를 사용한다.
*   **아이콘:** 필요한 경우 기능을 명확히 전달하는 간결한 아이콘을 사용한다.

## 7. 릴리즈 계획 (단계별 개발 계획)

*   **1단계: 환경 설정 및 API 연동 (완료 목표: Sprint 1)**
    *   Google AI Studio API 키 확보
    *   Flask 서버 기본 구조 설정
    *   Gemini API 연동 및 기본 프롬프트 변환 기능 테스트
*   **2단계: 프론트엔드 UI 개발 (완료 목표: Sprint 2)**
    *   키워드 입력창, 수신자 선택 UI, 결과창 UI 컴포넌트 개발
    *   반응형 CSS 적용
*   **3단계: 백엔드 로직 구현 (완료 목표: Sprint 3)**
    *   수신자(상사, 동료, 고객)별 최적화된 프롬프트 엔지니어링 설계
    *   `/convert` API 엔드포인트 개발 및 프론트엔드 연동
*   **4단계: 통합 및 배포 (완료 목표: Sprint 4)**
    *   Firebase 프로젝트 설정
    *   전체 기능 통합 테스트
    *   Firebase Hosting 및 Cloud Functions를 이용한 프로덕션 배포
